## Specification 

### Cryptographic aspects

*1. Cryptographic primitives*
| Function | Algorithm | Parameters |
| --- | --- | --- |
| `KDF from user_key` | Argon2id | m = 64 MiB, t = 3, p = 1|
| `KDF from master_key` | HKDF-SHA256 | --- |
| `AEAD` | ChaCha20-Poly1305 | --- |
| `MAC` | HMAC-SHA256 | --- |
| `RNG` | OsRng | --- |

Where:
m - memory cost
t - iterations
p - parallelism

*2. Key hierarchy*
user_key -> (Argon2id) -> master_key -> (HKDF-SHA256) -> wrap_key and header_mac_key
dek (random 32B) is encrypted by wrap_key.
dek encrypts body.
header_mac_key is used only to compute and verify the HMAC-SHA256 over the canonical file header.

*3. Header authentication*

The file header is stored in plaintext because it contains values required before decryption, such as Argon2id parameters, salt, AEAD identifier, nonce for DEK wrapping, and wrapped_DEK.
Although the header is not encrypted, it must be protected against tampering.
If any field of the canonical header is modified, the computed HMAC will not match the stored `header_mac`, and the vault MUST return a generic error.

*4. Nonce requirements*

ChaCha20-Poly1305 uses 12-byte nonces.
The implementation uses two types of nonces:
- `nonce_dek` for encrypting the `dek` with `wrap_key`,
- `nonce_body` for encrypting the vault body with `dek`.
A nonce MUST NOT be reused with the same key. Every body encryption MUST use a fresh `nonce_body` generated by `OsRng`.
Reusing the same `(key, nonce)` pair in ChaCha20-Poly1305 is a critical security error.

*5. Error handling*

ALL errors MUST be reported using the same generic error.
The implementation MUST NOT reveal whether the error was caused by an incorrect password or by a corrupted file.
Generic error:
`ERR_BAD_PASSWORD_OR_CORRUPTED`

*6. Key zeroization*

The following sensitive values MUST be zeroized after use:
- `user_key`,
- `master_key`,
- `wrap_key`,
- `header_mac_key`.
The `dek` is zeroized after losing the session or if it isn't of any use anymore.

*7. Password change*

Changing the master password does not change the `dek`.
During `changepass`, the old password is used to derive the old `wrap_key` and decrypt `wrapped_DEK`. Then the new password is used to derive a new `wrap_key` and `header_mac_key`.
The same `dek` is encrypted again with the new `wrap_key`, producing a new `wrapped_dek`. The header MAC is recomputed, and the body is re-encrypted or re-authenticated with the new header as AEAD associated data.

*8. HKDF separation*
Two subkeys are derived from master_key using HKDF-SHA256 with distinct info vaules:

wrap_key        ← HKDF(master_key, salt = none, info = "vault-v1wrap-dek-key")
header_mac_key  ← HKDF(master_key, salt = none, info = "vault-v1header-mac")

Using distinct info values ensures that 
wrap_key and header_mac_key are cryptographically independent, 
even though they are derived from the same master_key.

*9. AAD (Additional Authenticated Data)
AEAD encryption in this implementation uses AAD to bind 
ciphertext to its context and prevent substitution attacks.
Two AAD values are used:

- wrapped_dek:
   AAD = "vault-v1wrap-dek"
   This constant string identifies the purpose of the encryption
   and prevents the wrapped_dek from being used in a different context.
- ct_body:
   AAD = canonical_header || header_mac
   The entire canonical header (bytes 0..99) concatenated with 
   header_mac (bytes 100..131) is used as AAD for body encryption.
   This means any modification to the header fields (e.g. Argon2id 
   parameters, salt, wrapped_dek) will cause body decryption to fail,
   as the AAD will no longer match.
*10. Constant Time MAC comparison*
The comparison of expected_mac and stored header_mac MUST be 
performed in constant time.

Standard equality operators (==) MUST NOT be used for this comparison,
as they may terminate early upon finding the first differing byte,
leaking information about the compared values through response timing.

The implementation uses ConstantTimeEq from the subtle crate:

    expected_mac.ct_eq(header_mac)

This ensures that the comparison always takes the same amount of time
regardless of how many bytes match, eliminating the timing side-channel
on the authentication path.

### CLI — Command-Line Interface

**1. Runtime environment**
The tool is distributed as a single binary named `vault`. It requires no environment variables or configuration files. Top-level operations are invoked through subcommands:

| Subcommand | Arguments | Description |
| --- | --- | --- |
| `init` | `<path>` | Creates a new vault. The target file must not already exist. Prompts for a master password twice (with confirmation). |
| `open` | `<path>` | Verifies the master password, decrypts the vault and starts an **interactive session** (see §2). |
| `verify` | `<path> [--with-password]` | Without a password: checks file structure only. With `--with-password`: full cryptographic integrity check of header and body. |

**2. Interactive session model**
On success, `open` prints the file path, the record count and a hint, then enters a read-eval-print loop with the prompt `vault> `. The master key (`DEK`) is unwrapped once and **kept in memory for the lifetime of the session**; subsequent commands operate without re-entering the password. Each line is parsed as a subcommand; an empty line is ignored, EOF (Ctrl+Z / Ctrl+D) closes the session. Available session commands:

| Command | Arguments | Description |
| --- | --- | --- |
| `list` | `[--type=T] [--tag=X]` | Lists records (id, type, title, tags) — **never secret values** (F-03). Optional filters by type and/or tag. |
| `get` | `<id\|title> [--clip] [--field=F]` | Default: shows the full record including secrets. `--clip`: copies a field to the clipboard instead of showing it (F-04). `--field` without `--clip`: shows only that field. |
| `add` | `<type>` | Adds a record interactively (`login\|note\|apikey\|totp\|sshkey`). |
| `edit` | `<id\|title>` | Edits a record interactively; empty input keeps the current value. |
| `rm` | `<id\|title>` | Removes a record after confirmation. |
| `changepass` | — | Changes the master password (re-wraps the DEK; body unchanged). |
| `upgrade-kdf` | `[--memory] [--iterations] [--parallelism]` | Strengthens Argon2id parameters (re-wraps the DEK under the same password). |
| `attach` | `<file> [--title]` | Adds a file (≤ 5 MiB) as an `attachment` record. |
| `extract` | `<id> [<dest>]` | Writes an attachment's bytes to a file. |
| `help` | — | Prints the list of session commands. |
| `exit` / `quit` | — | Closes the session. |

After every mutating command (`add`, `edit`, `rm`, `changepass`, `upgrade-kdf`, `attach`) the vault is **saved automatically** (see §13). A failure of a single command prints an error and returns to the prompt without ending the session.

**3. Password input (F-19)**
The master password is always read interactively from the terminal without echo, using `rpassword::prompt_password`. This applies to `init`, `open`, `verify --with-password`, `changepass`, `upgrade-kdf`, and to every secret record field (e.g. `password`, `private_key`, TOTP secret) entered via `add` / `edit`. The password (and any secret field) is never:

- passed as a command-line argument,
- read from an environment variable,
- logged or written to stdout/stderr.

On `init` and on the new password in `changepass`, the password is read twice and compared using a constant-time routine. On `open`, `verify --with-password` and the old password in `changepass`, it is read once without confirmation.

**4. Constant-time password comparison**
Whenever two password inputs are confirmed (`init`, new password in `changepass`), they are compared by this scheme:

```
if len(p1) != len(p2): reject
diff = 0
for each byte pair (a, b): diff |= a XOR b
if diff != 0: reject
```

This avoids leaking information through timing during the confirmation step.

**5. Error handling (F-17)**
Any error encountered while opening a vault — in `open` and in `verify --with-password` — is returned as the **single unified error message**, regardless of whether the cause was a wrong password, a corrupted/truncated file, a header-MAC mismatch, a failed AEAD tag, or malformed body. `verify` **without** a password is the exception: it reports specific structural errors (bad magic, unsupported version, wrong field lengths, etc.), because it performs no cryptographic check.

**6. Key material zeroization (F-16)**
Password buffers read from the terminal are stored as `Zeroizing<String>` and zeroed automatically when they go out of scope. The session holds the unwrapped `DEK` as `Zeroizing<[u8; 32]>`; it is zeroed when the session ends (on `exit`, EOF, or `lock`). The master password itself is **not** retained for the session — only the DEK is.

**7. Validation order for `init`**
  1. Check that the target file does not exist — if it does, return an error immediately **without** prompting for a password.
  2. Read the password with confirmation.
  3. Reject an empty password.
  4. Write the vault by calling `save_vault` (writes a `VLT1` file with an empty record list).

**8. Validation order for `open`**
  1. Check that the target file exists — if not, return an error immediately **without** prompting for a password.
  2. Read the password once, without confirmation.
  3. Read and parse the file, derive keys from the password (Argon2id → HKDF), verify the header MAC, unwrap the DEK and decrypt the body (AEAD). Any failure in this step returns the single unified error message (§5).
  4. Parse the decrypted body (CBOR). The DEK is **retained in memory** and the interactive session starts.

**9. Top-level command `verify`**
  - **Without `--with-password`** — parses the header only and checks: magic `VLT1`, version, reserved flags, KDF id and non-zero parameters, salt length, AEAD id, wrapped-DEK length and minimum file length. Returns a specific structural error on failure. Does **not** prove integrity.
  - **With `--with-password`** — prompts for the password and performs the full open path (header-MAC verification + DEK unwrap + body AEAD decryption + CBOR parse) **without** starting a session. Reports success or the single unified error message (§5).

**10. Record identification**
A record is selected by **exact title** (case-insensitive) or by a **UUID prefix** in hexadecimal (hyphens ignored). Zero matches → "not found"; more than one match → an "ambiguous, give a longer prefix" error. No mutation is performed on an ambiguous selection.

**11. Record types and field validation (per §4.2)**
Per-type fields are prompted in `add` / `edit`. Secret fields are read without echo. Type-specific constraints:

- `totp.algorithm` ∈ {`SHA1`, `SHA256`, `SHA512`} (default `SHA1`)
- `totp.digits` ∈ {`6`, `8`} (default `6`)
- `totp.period` ∈ {`30`, `60`} (default `30`)
- `totp.secret_base32` — validated as Base32 (RFC 4648 alphabet `A–Z`, `2–7`, optional `=`; whitespace stripped, upper-cased)

In `add`, an empty value for a constrained field uses its default; an empty value for a free field leaves it unset. In `edit`, an empty value keeps the current one. `attachment` records cannot be created with `add` — use `attach`.

**12. Attachments**
`attach <file>` reads the file (rejecting anything larger than **5 MiB = 5 242 880 bytes**) and stores it as an `attachment` record with fields `filename`, `mime_type` (guessed from the extension, default `application/octet-stream`), `size` and `data` (raw bytes). `extract <id> [<dest>]` writes the `data` bytes to `dest`; if `dest` is omitted the original filename in the current directory is used, if `dest` is a directory the original filename is appended, and an existing target file is never overwritten.

**13. Persistence model**
The session keeps the decrypted records in memory. On save, the body is re-serialized (CBOR) and re-encrypted with the **same DEK** and the **same header**, generating only a **new `nonce_body`** — Argon2id is **not** re-run (see §7.5). `changepass` and `upgrade-kdf` instead rebuild the header (new salt, freshly wrapped DEK, new header MAC) and re-encrypt the body with the new AAD (see §7.6 / §7.7), keeping the same DEK. Every write is atomic: write-temp + fsync + rename (F-20).

**14. Clipboard handling (F-04 / F-18)**
`get --clip` copies the selected field (the primary secret of the record, or `--field=F`) to the system clipboard (on Windows via `clip.exe`). After 30 seconds the clipboard is cleared **only if it still contains the copied value**, so a value the user has since replaced is left untouched.

**15. Exit codes**
Top-level commands return `0` on success and `1` on failure (e.g. file already exists for `init`, wrong password / corrupted file, structural error in `verify`). Errors inside an open session are reported to the user and return to the prompt without changing the process exit code; the session ultimately exits with `0`.

**16. Export / import (F-10 / F-11)**

Two session commands move records in and out of the vault as **plaintext CBOR** — the same CBOR encoding used for the encrypted body, but written unencrypted. CBOR (rather than JSON) was chosen so that no serialization library outside the approved set is required: the record structures already round-trip through `ciborium`.

| Command | Arguments | Description |
| --- | --- | --- |
| `export` | `<file> [--format=cbor]` | Writes all records to a plaintext CBOR file. |
| `import` | `<file> [--format=cbor]` | Reads records from a CBOR file and appends them to the open vault. |

- **Format flag** — `--format` accepts only `cbor` (the default); any other value is rejected with an error. This keeps the door open for additional formats without changing the interface.
- **`export`**:
  1. Refuses to run if the target file already exists.
  2. Prints a **loud warning** that the output is an unencrypted, cleartext file containing every secret, and requires an explicit `[t/N]` confirmation before writing (F-10).
  3. Serializes the current body (`schema_version` + records) with the same CBOR encoder used for the encrypted body, then writes it to the file in cleartext. The resulting file is byte-identical to the vault's decrypted body.
- **`import`**:
  1. Reads and CBOR-decodes the file into a record set; a malformed or non-CBOR file yields a single error.
  2. Appends the imported records to the open vault, **skipping any record whose `id` (UUID) already exists** — re-importing the same file is therefore idempotent.
  3. Persists through the normal save path (§13) and reports how many records were added versus skipped.
- **Security note** — the export file is deliberately **unencrypted** (a cleartext dump intended for backup or migration, per F-10). Anyone with access to it can read every secret; it must be treated like a plaintext password list and deleted after use. The encrypted vault itself is never weakened by these commands.

**Format & Storage aspects**
1. File structure
The vault file consists of four consecutive sections with no padding:
offset    size    field
0         100B    canonical header
100        32B    header_mac
132        12B    nonce_body
144         NB    ct_body
Minimum valid file size is 144 bytes. Files shorter than 144 bytes MUST be rejected before any cryptographic operation.
2. Canonical header layout
The canonical header is exactly 100 bytes, big-endian:
offset    size    field              value
0          4B     magic              "VLT1"
4          2B     version            0x0001
6          2B     flags              0x0000 (reserved, MUST be zero)
8          1B     kdf_id             0x01 = Argon2id
9          4B     kdf_memory_kib     e.g. 65536
13         4B     kdf_iterations     e.g. 3
17         1B     kdf_parallelism    e.g. 1
18         1B     kdf_salt_len       16
19        16B     kdf_salt           random
35         1B     aead_id            0x01 = ChaCha20-Poly1305
36        12B     nonce_dek          random
48         4B     wrapped_dek_len    48
52        48B     wrapped_dek        AEAD(wrap_key, nonce_dek, dek)
Header size MUST be exactly 100 bytes. This is enforced by a compile-time assert in to_bytes().
3. Record schema
Every record contains the following fields:
field         type              description
id            bytes(16)         UUID v4, immutable
type          string            login, note, apikey, totp, sshkey, attachment
title         string            user label
tags          string[]          list of tags
notes         string            free-form text
created_at    uint              Unix nanoseconds
modified_at   uint              Unix nanoseconds
fields        map(str→bytes)    type-specific fields
4. Body serialization
The vault body is serialized using CBOR. CBOR is used because it is binary, compact, and natively supports byte strings — required for binary field values and attachment data.
5. Atomic write
Every write MUST follow this sequence:

Write all data to vault.tmp
Call fsync on the temporary file
Call rename(vault.tmp, vault.db)

If the process is interrupted before step 3, the original file remains intact.
6. Structural verification (without password)
verify_structure() checks the file without a password, in this order:

file size ≥ 144 bytes
magic == "VLT1"
version == 1
flags == 0x0000
kdf_id == 1
kdf_memory_kib > 0
kdf_iterations > 0
kdf_parallelism > 0
kdf_salt_len == 16
aead_id == 1
wrapped_dek_len == 48
7. Error handling
Structural errors return a typed VerifyError with a distinct variant per failed check, allowing specific diagnostic messages.
8. Fuzzing targets
Two public functions are exposed for the fuzzer:
fuzz_parse_header(data: &[u8]) — takes raw bytes and runs header verification on them directly, without reading any file.
fuzz_parse_body(data: &[u8]) — takes raw bytes and tries to parse them as CBOR vault body. Returns None if parsing fails.
